# --interactive--

L'attribut `aria-describedby` est utilisé pour fournir des informations supplémentaires sur un élément aux utilisateurs de lecteurs d'écran en référençant du contenu existant sur la page. Il crée une association programmatique entre l'élément et le contenu (techniquement appelée description accessible), que les lecteurs d'écran peuvent utiliser pour informer les utilisateurs des informations supplémentaires quand ils interagissent avec l'élément. 

L'utilisation la plus courante de `aria-describedby` est d'associer des instructions et des messages d'erreur avec des champs de formulaire. À cause des diverses méthodes que les utilisateurs de lecteurs d'écran ont pour naviguer dans une page, ils peuvent rater ces messages en passant d'un champ à un autre. Utiliser `aria-describedby` aide à garantir qu'ils les entendront.

Regardons quelques exemples pour comprendre comment ça fonctionne. Dans ce premier exemple, on a un élément `form` qui accepte un mot de passe.

Active l'éditeur interactif et tape quelques caractères dans le champ de saisie du mot de passe. Tu verras que le mot de passe est masqué dans la fenêtre d'aperçu. Tu verras aussi que le texte `password-help` reste rouge jusqu'à ce que tu fournisses 8 caractères ou plus dans le champ. 

NOTE : Cet exemple interactif utilise du CSS et du JavaScript pour mettre à jour dynamiquement la couleur du texte `password-help`. Ne t'inquiète pas pour la compréhension du code JavaScript car tu apprendras le JavaScript dans des modules futurs.

:::interactive_editor

```html
<link rel="stylesheet" href="styles.css">

<form>
  <label for="password">Password:</label>
  <input type="password" id="password" aria-describedby="password-help" />
  <p id="password-help">Your password must be at least 8 characters long.</p>
</form>

<script src="index.js"></script>
```

```css
#password-help {
  color: red;
}
```

```js
const passwordEl = document.getElementById("password");
const passwordHelpText = document.getElementById("password-help");

passwordEl.addEventListener("input", (e) => { 
  const userInput = e.target.value;
  passwordHelpText.style.color = userInput.length >= 8 ? "green" : "red";
});
```

:::

On utilise un élément `label` pour le texte `Password` et on l'associe avec le champ de saisie de mot de passe.

On a aussi un élément paragraphe qui décrit les exigences du mot de passe. On utilise l'attribut `aria-describedby` pour associer le champ de saisie de mot de passe avec les exigences du mot de passe dans l'élément paragraphe. Quand un utilisateur de lecteur d'écran interagit avec ce champ, son lecteur d'écran annoncera le nom du champ (`Password`) puis peut-être aussi les exigences du mot de passe. Cependant ce n'est pas une garantie absolue car certains lecteurs d'écran peuvent ne pas annoncer automatiquement le contenu supplémentaire, ou peuvent ne l'annoncer que dans des circonstances spécifiques. Ce n'est pas rare. Chaque lecteur d'écran est différent et gère les attributs ARIA à sa manière. Cela ne remet pas en cause l'utilisation de `aria-describedby`, car cela profitera aux utilisateurs de lecteurs d'écran en général.

Un autre bon cas d'usage pour l'attribut `aria-describedby` est quand tu as un bouton de suppression. Voici un exemple d'un bouton de suppression suivi d'un message décrivant ce qui se passera quand le bouton sera cliqué :

:::interactive_editor

```html
<button aria-describedby="delete-message">Delete</button>

<p id="delete-message">Warning! All deletions are permanent.</p>
```

:::

Tout comme dans l'exemple précédent, on associe le bouton de suppression au message en utilisant l'attribut `aria-describedby`. La valeur de l'`id` et la valeur de l'attribut `aria-describedby` doivent correspondre. 

L'attribut `aria-describedby` est un attribut puissant qui peut être utilisé pour s'assurer que des informations supplémentaires sur un élément sont fournies aux utilisateurs de lecteurs d'écran quand ils interagissent avec l'élément. Il est le plus couramment utilisé pour associer des instructions et des messages d'erreur avec des champs de formulaire afin de réduire le risque que les utilisateurs de lecteurs d'écran ratent ces messages en naviguant dans le formulaire.
