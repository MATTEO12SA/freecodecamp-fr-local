# --interactive--

Rendre du contenu statique accessible peut être relativement simple, mais le contenu dynamique peut être plus difficile. C'est là que WAI-ARIA entre en jeu.

Voyons ce qu'est WAI-ARIA, son but, comment il fonctionne et quelques exemples.

WAI-ARIA signifie Web Accessibility Initiative - Accessible Rich Internet Applications. C'est une spécification qui améliore l'accessibilité du contenu dynamique et des composants d'interface utilisateur (UI).

Note que WCAG et WAI-ARIA ne sont pas la même chose. WCAG fournit des recommandations générales pour l'accessibilité web, alors que WAI-ARIA propose des règles spécifiques pour rendre le contenu dynamique et interactif accessible aux utilisateurs de technologies d'assistance.

Donc, le but principal de WAI-ARIA est d'améliorer l'accessibilité pour le contenu dynamique et les composants d'interface qui n'ont pas d'équivalents HTML natifs.

WAI-ARIA fonctionne en introduisant un ensemble d'attributs que tu peux ajouter aux éléments HTML pour fournir des informations sémantiques supplémentaires. Ces attributs sont classés en rôles, états et propriétés.

Le rôle ARIA définit le but d'un élément dans un site web ou une application web. Voici un exemple où l'on définit le rôle à `button` pour un élément `div`.

```html
<div role="button">Click Me</div>
```

En faisant cela, tu indiques à la technologie d'assistance que l'élément est un bouton. Cependant, les rôles ne fournissent aucune fonctionnalité. Le simple fait de donner à ce `div` un `role` de `button` ne le fera pas se comporter comme un bouton. Pour qu'il ressemble à un bouton et se comporte comme tel, tu devras utiliser du CSS et du JavaScript pour obtenir le résultat souhaité. 

Voici un exemple d'utilisation de HTML, CSS et JavaScript pour créer un élément `button` personnalisé. 

**NOTE** : Ne t'inquiète pas pour la compréhension du code CSS et JavaScript. Tu apprendras ces langages dans des modules futurs. 

:::interactive_editor

```html
<link href="styles.css" rel="stylesheet">
<div id="custom-btn" role="button">Click Me</div>
<script src="index.js"></script>
```

```css
#custom-btn {
  display: inline-block;
  padding: 0.4em 1em;
  font-size: 1rem;
  font-family: sans-serif;
  color: buttontext;
  background-color: buttonface;
  border: 1px solid #ccc;
  border-radius: 4px;
  cursor: pointer;
  user-select: none;
  text-align: center;
}

#custom-btn:focus {
  outline: 2px solid Highlight;
  outline-offset: 2px;
}

#custom-btn:active {
  background-color: #ddd;
}
```

```js
const button = document.getElementById("custom-btn");

button.addEventListener("click", () => {
  alert("Button clicked!");
});

button.addEventListener("keydown", (e) => {
  if (e.key === "Enter" || e.key === " ") {
    e.preventDefault(); 
    button.click();
  }
});
```

:::

Il est toujours préférable d'utiliser l'élément natif `button` ou `input` avec `type="button"` à la place.

Les propriétés ARIA fournissent des détails supplémentaires sur les éléments. Par exemple, la propriété `aria-labelledby` te permet de connecter un élément à un libellé spécifique :

```html
<h2 id="header-id">About freeCodeCamp</h2>
<button id="button-id" aria-labelledby="header-id button-id">Learn More</button>
```

Cela rendra les éléments compréhensibles et navigables pour les utilisateurs de technologies d'assistance.

Pour tirer le meilleur parti de WAI-ARIA, essaie de t'en tenir au HTML natif autant que possible, car il fournit généralement plus d'accessibilité dès le départ.

Utilise WAI-ARIA uniquement quand le HTML est insuffisant, et n'oublie pas de tester avec des technologies d'assistance comme les lecteurs d'écran, ou de faire tester ton travail par des personnes en situation de handicap. Assure-toi aussi que tes états et propriétés WAI-ARIA se mettent à jour avec le contenu en temps réel. Évite d'abuser d'ARIA, car cela peut souvent prêter à confusion.
