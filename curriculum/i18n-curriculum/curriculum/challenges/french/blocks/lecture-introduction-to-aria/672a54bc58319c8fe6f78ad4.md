# --interactive--

Si tu as un jour besoin d'afficher du contenu tout en le cachant aux personnes qui utilisent des technologies d'assistance, comme les lecteurs d'écran, tu peux utiliser l'attribut `aria-hidden`.

Tu n'as qu'à l'ajouter à l'élément HTML que tu veux cacher et lui donner la valeur `true`, comme tu peux le voir ici : `aria-hidden="true"`.

Cet attribut cache l'élément et tous ses enfants de l'arbre d'accessibilité, tout en les gardant visibles sur la page. Les cas d'utilisation courants incluent :

- Les icônes et les images qui n'ont qu'un but décoratif.
- Le contenu dupliqué.

Il est important de retenir que `aria-hidden` ne cache le contenu qu'aux technologies d'assistance, comme les lecteurs d'écran. Si le contenu doit être caché à tout le monde, alors tu ne devrais pas utiliser `aria-hidden` pour le cacher. Par exemple, un menu hamburger qui est actuellement réduit doit être caché à tous les utilisateurs de clavier, pas seulement aux utilisateurs de lecteurs d'écran. Dans ce cas, tu pourrais définir la propriété CSS `display` à `none` sur le menu pour le retirer du DOM quand il est réduit.

Tu ne devrais jamais utiliser `aria-hidden` pour cacher un élément qui peut recevoir le focus au clavier. L'attribut `aria-hidden` ne retire l'élément que de l'arbre d'accessibilité. Il ne le retire pas du DOM. Ainsi, il restera possible pour les utilisateurs de lecteurs d'écran de naviguer avec Tab jusqu'à l'élément, mais leur lecteur d'écran n'annoncera pas l'élément, ce qui les amènera à se concentrer sur « rien ».

Voici un exemple où l'on cache une icône de l'arbre d'accessibilité en ajoutant l'attribut `aria-hidden` avec la valeur `true`. 

On ne garde que le texte exposé aux technologies d'assistance pour éviter toute confusion qui pourrait découler de la redondance d'avoir à la fois une icône et un texte pour le même usage.

**NOTE** : Cet exemple interactif inclut le CDN Font Awesome pour que tu puisses voir l'icône engrenage affichée dans la fenêtre d'aperçu. Pour voir les aperçus, tu devras activer l'éditeur interactif.

:::interactive_editor

```html
<head>
  <!-- Font Awesome CDN -->
  <link
    rel="stylesheet"
    href="https://cdn.jsdelivr.net/npm/@fortawesome/fontawesome-free@6.5.0/css/all.min.css"
  />
</head>

<button>
  <i class="fa-solid fa-gear" aria-hidden="true"></i>
  <span class="label">Settings</span>
</button>
```

:::

Tu n'as pas besoin d'utiliser `aria-hidden` quand :

- L'élément HTML a déjà un attribut `hidden`.
- L'élément ou un ancêtre de l'élément est déjà caché avec `display: none`.
- L'élément ou un ancêtre de l'élément est déjà caché avec `visibility: hidden`.

Dans ces trois cas, l'élément est déjà retiré du DOM et donc caché de l'arbre d'accessibilité, donc l'attribut `aria-hidden` n'est pas nécessaire.

Comme pour l'utilisation de n'importe quel attribut ARIA, tu devrais toujours tester avec des technologies d'assistance, et de préférence faire tester ton travail par des personnes en situation de handicap, pour t'assurer que c'est facile à comprendre, même avec ces éléments cachés.

Tu devrais aussi savoir que mettre `aria-hidden` à `false` n'exposera pas l'élément aux technologies d'assistance si l'un de ses parents a cet attribut mis à `true`.

L'attribut `aria-hidden` est utilisé pour cacher des éléments aux personnes qui utilisent des technologies d'assistance, comme les utilisateurs de lecteurs d'écran.

Bien qu'il puisse être utile pour cacher des éléments purement décoratifs et du contenu dupliqué, il devrait être utilisé avec parcimonie pour éviter d'entraver l'accessibilité.

En général, tout le contenu et toutes les fonctionnalités disponibles sur la page devraient aussi être disponibles pour les personnes utilisant des technologies d'assistance. Le cas d'usage pour `aria-hidden` est très étroit et devrait se limiter principalement à rendre l'expérience plus propre pour les utilisateurs de lecteurs d'écran en supprimant les informations purement décoratives ou dupliquées. N'utilise pas `aria-hidden` pour cacher du contenu dont tu penses que les utilisateurs de lecteurs d'écran ne seraient pas intéressés. Les utilisateurs de lecteurs d'écran méritent d'avoir accès à toutes les informations de la page.

En suivant ces bonnes pratiques et en testant l'expérience utilisateur, tu peux créer des expériences en ligne inclusives pour tout le monde.
